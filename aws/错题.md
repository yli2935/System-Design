## 题目 1

**题干：** A customer needs to organize and consolidate information based on criteria specified in tags or resources in AWS. Which of the following services would you recommend to satisfy this requirement?

**选项：**
- A. AWS Directory Service
- B. Amazon CloudWatch Dashboard
- C. AWS Resource Groups ✅
- D. AWS IAM Group

**正确答案：** C. AWS Resource Groups

### 知识点总结

**AWS Resource Groups（资源组）：**
- 允许根据**标签（Tags）**或 **AWS CloudFormation 堆栈**来组织和管理 AWS 资源
- 可以将具有相同标签的资源分组在一起，便于统一查看、管理和执行批量操作
- 支持跨多种 AWS 服务的资源整合
- 常见用途：按项目、环境（dev/prod）、部门、成本中心等维度组织资源

**其他选项分析：**
| 服务 | 用途 |
|------|------|
| AWS Directory Service | 提供托管的 Microsoft Active Directory 或简单目录服务，用于身份管理 |
| Amazon CloudWatch Dashboard | 用于监控指标的可视化仪表板，不是资源组织工具 |
| AWS IAM Group | 用于管理 IAM 用户的权限分组，不是资源组织工具 |

**关键词：** "organize"、"consolidate"、"tags" → 想到 **Resource Groups**

## 题目 2

**题干：** A company plans to use an application streaming service to give its employees instant access to their desktop applications from any device. Which of the following services fulfills this requirement?

**选项：**
- A. Amazon AppStream 2.0 ✅
- B. AWS AppSync
- C. Amazon WorkSpaces
- D. Amazon Kinesis Data Streams

**正确答案：** A. Amazon AppStream 2.0

### 知识点总结

**Amazon AppStream 2.0：**
- 完全托管的**应用程序流式传输服务**
- 将桌面应用程序流式传输到用户的 Web 浏览器（无需安装）
- 用户可以从**任何设备**访问应用程序
- 应用程序在 AWS 上运行，只有像素被流式传输到用户设备
- 适用场景：软件试用、培训、远程办公等

**AppStream 2.0 vs WorkSpaces 区别：**
| 服务 | 类型 | 说明 |
|------|------|------|
| **AppStream 2.0** | 应用程序流 | 流式传输**特定应用程序**，用户只能访问指定的 App |
| **WorkSpaces** | 虚拟桌面 (VDI) | 提供**完整的云桌面环境**，用户拥有整个 Windows/Linux 桌面 |

**其他选项分析：**
| 服务 | 用途 |
|------|------|
| AWS AppSync | 用于构建 GraphQL API 的托管服务（名字容易混淆！） |
| Amazon Kinesis Data Streams | 实时数据流处理服务，用于收集和处理大规模流数据 |

**关键词：** "application streaming" → 想到 **AppStream 2.0**；"virtual desktop" → 想到 **WorkSpaces**

## 题目 3

**题干：** A gaming company needs a service that uses the AWS global network to optimize users' access speed to their applications through an anycast static IP address. Which of the following services fits this criteria?

**选项：**
- A. AWS Global Accelerator ✅
- B. Amazon Route 53
- C. Amazon CloudFront
- D. Amazon ElastiCache

**正确答案：** A. AWS Global Accelerator

### 知识点总结

**AWS Global Accelerator：**
- 使用 **AWS 全球网络**（而非公共互联网）来优化流量路由
- 提供 **2 个 Anycast 静态 IP 地址**，作为应用程序的固定入口点
- 将用户流量引导至最近的 AWS 边缘节点，然后通过 AWS 骨干网传输
- 支持 **TCP 和 UDP** 协议
- 适用场景：游戏、IoT、VoIP、视频会议等需要低延迟的非 HTTP 应用

**Global Accelerator vs CloudFront 区别（重要！）：**
| 特性 | Global Accelerator | CloudFront |
|------|-------------------|------------|
| 类型 | 网络层加速 | CDN（内容分发） |
| 协议 | TCP / UDP | HTTP / HTTPS |
| 静态 IP | ✅ 提供 Anycast 静态 IP | ❌ 无静态 IP |
| 缓存 | ❌ 不缓存 | ✅ 缓存静态内容 |
| 适用场景 | 游戏、实时通信、非 HTTP 应用 | 网站、视频流、静态资源 |

**其他选项分析：**
| 服务 | 用途 |
|------|------|
| Amazon Route 53 | DNS 服务，支持地理位置路由，但不提供静态 IP 加速 |
| Amazon ElastiCache | 内存缓存服务（Redis/Memcached），用于数据库缓存 |

**关键词：** "global network" + "anycast static IP" + "gaming" → 想到 **Global Accelerator**

## 题目 4

**题干：** Among the following services, which is the most suitable one to use to store the results of I/O-intensive SQL database queries to improve application performance?

**选项：**
- A. Amazon ElastiCache ✅
- B. AWS Greengrass
- C. AWS IoT Greengrass
- D. Amazon DynamoDB Accelerator (DAX)

**正确答案：** A. Amazon ElastiCache

### 知识点总结

**Amazon ElastiCache：**
- 完全托管的**内存缓存服务**
- 支持两种引擎：**Redis** 和 **Memcached**
- 可缓存**任何数据源**的查询结果，包括 SQL 数据库（RDS、Aurora 等）
- 将热点数据存储在内存中，微秒级响应，大幅减少数据库 I/O 压力

**ElastiCache vs DAX 区别（易错点！）：**
| 特性 | ElastiCache | DAX |
|------|-------------|-----|
| 适用数据库 | **任何数据库**（SQL / NoSQL） | **仅 DynamoDB** |
| 缓存类型 | 通用内存缓存 | DynamoDB 专用缓存 |
| 使用场景 | RDS、Aurora、自建数据库等 | 只能加速 DynamoDB 查询 |

**题目陷阱分析：**
- 题目说的是 **"SQL database"**，所以 DAX 不适用（DAX 只能用于 DynamoDB，这是 NoSQL）
- 如果题目说的是 "DynamoDB queries"，那答案就是 DAX

**其他选项分析：**
| 服务 | 用途 |
|------|------|
| AWS Greengrass / IoT Greengrass | IoT 边缘计算服务，让设备在本地运行 Lambda，与缓存无关 |

**关键词判断：**
- "SQL database" + "cache" → **ElastiCache**
- "DynamoDB" + "cache" → **DAX**

## 题目 5

**题干：** Which AWS service helps identify resources shared with an external entity and generates policies based on access activity?

**选项：**
- A. AWS License Manager
- B. AWS Systems Manager Parameter Store
- C. AWS Proton
- D. AWS IAM Access Analyzer ✅

**正确答案：** D. AWS IAM Access Analyzer

### 知识点总结

**AWS IAM Access Analyzer：**
- **核心功能 1：** 识别与**外部实体**共享的资源（如 S3 桶、IAM 角色、KMS 密钥、Lambda 函数、SQS 队列等）
- **核心功能 2：** 基于 CloudTrail 日志中的**访问活动**自动生成**最小权限策略**
- 帮助发现意外的公开访问或跨账户访问
- 支持持续监控，发现问题时生成 findings（发现）

**IAM Access Analyzer 的两大用途：**
| 功能 | 说明 |
|------|------|
| 外部访问分析 | 识别哪些资源被外部 AWS 账户或公共互联网访问 |
| 策略生成 | 根据实际使用情况生成最小权限 IAM 策略 |

**其他选项分析：**
| 服务 | 用途 |
|------|------|
| AWS License Manager | 管理软件许可证（如 Windows、Oracle 等），跟踪许可证使用情况 |
| AWS Systems Manager Parameter Store | 存储配置数据、字符串、密钥等参数，集中管理配置 |
| AWS Proton | 用于容器和无服务器应用的**基础设施自动化部署**服务 |

**关键词：** "external entity" + "shared resources" + "generate policies" + "access activity" → **IAM Access Analyzer**

## 题目 6

**题干：** Which AWS service enables you to build event-driven applications and decouple the components of your application architecture?

**选项：**
- A. Amazon SQS
- B. Amazon Kinesis
- C. Amazon SNS
- D. Amazon EventBridge ✅

**正确答案：** D. Amazon EventBridge

### 知识点总结

**Amazon EventBridge：**
- **无服务器事件总线**，专为构建**事件驱动架构**设计
- 可接收来自 AWS 服务、自定义应用、SaaS 应用（如 Zendesk、Shopify）的事件
- 支持**事件模式匹配和过滤**，将事件路由到目标（Lambda、SQS、SNS、Step Functions 等）
- 使用 Schema Registry 自动发现和存储事件结构
- 核心用途：**松耦合、事件驱动的微服务架构**

**四大消息/事件服务对比（重要！）：**
| 服务 | 类型 | 核心特点 | 适用场景 |
|------|------|----------|----------|
| **EventBridge** | 事件总线 | 事件驱动架构、多源多目标、模式匹配 | 微服务事件路由、SaaS 集成 |
| **SQS** | 消息队列 | 点对点、保证投递、可持久化 | 任务队列、解耦生产者消费者 |
| **SNS** | 发布/订阅 | 一对多推送通知 | 通知、广播消息、移动推送 |
| **Kinesis** | 数据流 | 实时大规模数据流处理 | 日志收集、点击流、IoT 数据 |

**题目关键词分析：**
- "event-driven applications" → 直接指向 **EventBridge**（事件驱动的核心服务）
- "decouple" → SQS 也能解耦，但 EventBridge 更强调事件驱动

**记忆口诀：**
- 事件驱动 → **EventBridge**
- 消息队列 → **SQS**
- 广播通知 → **SNS**
- 实时数据流 → **Kinesis**

## 题目 7

**题干：** A company is using multiple AWS services to host their application, and they want to ensure that they optimize the environment by adhering to AWS best practices. Which of the following services is capable of inspecting your AWS environment and making recommendations to lower expenditures, improve system performance and reliability, and close security gaps?

**选项：**
- A. AWS Trusted Advisor ✅
- B. AWS Budgets
- C. AWS Cost Explorer
- D. AWS Inspector

**正确答案：** A. AWS Trusted Advisor

### 知识点总结

**AWS Trusted Advisor：**
- 提供 AWS **最佳实践建议**的服务
- 自动检查 AWS 环境，并在**五个类别**中提出建议：

| 支柱 | 英文 | 说明 |
|------|------|------|
| 成本优化 | Cost Optimization | 识别未使用资源、节省成本的机会 |
| 性能 | Performance | 提高应用程序性能的建议 |
| 安全性 | Security | 关闭安全漏洞、加强安全配置 |
| 容错性 | Fault Tolerance | 提高可用性和冗余的建议 |
| 服务限制 | Service Limits | 检查是否接近服务配额限制 |

**题目关键词对应：**
- "lower expenditures" → 成本优化 ✓
- "improve performance" → 性能 ✓
- "improve reliability" → 容错性 ✓
- "close security gaps" → 安全性 ✓

**易混淆服务对比：**
| 服务 | 用途 | 范围 |
|------|------|------|
| **Trusted Advisor** | 最佳实践建议 | 成本 + 性能 + 安全 + 容错 + 限制 |
| **AWS Inspector** | 安全漏洞扫描 | **仅安全**（EC2、容器的 CVE 漏洞） |
| **AWS Cost Explorer** | 成本分析可视化 | **仅成本**分析，无优化建议 |
| **AWS Budgets** | 预算警报 | **仅成本**警报，无优化建议 |

**记忆技巧：** 
- 题目同时提到**成本、性能、安全、可靠性** → 只有 **Trusted Advisor** 全覆盖
- Inspector 只管安全，Cost Explorer/Budgets 只管成本

## 题目 8

**题干：** Which of the following can you use to resolve the connection between your on-premises VPN and your Amazon VPC? (Select TWO)

**选项：**
- A. Virtual Private Gateway ✅
- B. Amazon Route 53 ✅
- C. Egress-Only Internet Gateway
- D. VPC Peering
- E. NAT Gateway

**正确答案：** A. Virtual Private Gateway + B. Amazon Route 53

### 知识点总结

**题目关键词 "resolve" 的双重含义：**
1. **建立连接（establish）** → Virtual Private Gateway
2. **DNS 解析（resolve DNS）** → Route 53 (Resolver)

**答案 A - Virtual Private Gateway (VGW)：**
- AWS 侧的 VPN 端点，附加到 VPC
- 用于建立 Site-to-Site VPN 连接
- 是混合云网络连接的必需组件

**答案 B - Amazon Route 53（Resolver）：**
- **Route 53 Resolver Endpoints** 实现本地与 VPC 之间的 DNS 解析
- **Inbound Endpoint：** 允许本地 DNS 查询 VPC 中的私有域名
- **Outbound Endpoint：** 允许 VPC 资源查询本地 DNS 服务器
- 混合云环境中，DNS 解析是连接正常工作的关键

```
混合云连接完整架构：
[本地数据中心] <==== VPN Tunnel ====> [Virtual Private Gateway] <---> [VPC]
     |                                                                  |
[本地 DNS] <==== Route 53 Resolver Endpoints ====> [VPC 私有 DNS]
```

**各选项分析：**
| 选项 | 用途 | 与混合云连接关系 |
|------|------|----------------|
| **Virtual Private Gateway** | VPN/Direct Connect 的 AWS 侧端点 | ✅ 建立网络连接 |
| **Amazon Route 53** | DNS 服务 + Resolver | ✅ 解析跨环境域名 |
| Egress-Only Internet Gateway | IPv6 出站流量网关 | ❌ 仅用于 IPv6 出站 |
| VPC Peering | VPC 之间的私有连接 | ❌ 用于 VPC 互联 |
| NAT Gateway | 私有子网访问互联网 | ❌ 用于 NAT 转换 |

**易错点：** 不要忽视 "resolve" 这个词的 DNS 解析含义！混合云环境不仅需要网络连通（VGW），还需要 DNS 解析（Route 53 Resolver）

## 题目 9

**题干：** A company is planning to launch a new system in AWS but they do not have an employee who has AWS-related expertise. Which of the following AWS channels can instead help the company design, architect, build, migrate, and manage their workloads and applications on AWS?

**选项：**
- A. AWS Partner Network Consulting Partners ✅
- B. AWS Marketplace
- C. AWS Partner Network Technology Partners
- D. Technical Account Management

**正确答案：** A. AWS Partner Network Consulting Partners

### 知识点总结

**AWS Partner Network (APN) 两类合作伙伴（重要区分！）：**

| 类型 | 提供内容 | 适用场景 |
|------|----------|----------|
| **Consulting Partners（咨询合作伙伴）** | 专业**服务** | 帮客户设计、架构、构建、迁移、管理 AWS 工作负载 |
| **Technology Partners（技术合作伙伴）** | 软件**产品** | 提供托管在 AWS 或与 AWS 集成的软件解决方案 |

**题目关键词分析：**
- "design, architect, build, migrate, manage" → 这些都是**服务**，不是产品
- 需要**人**来帮忙做事 → **Consulting Partners**

**各选项分析：**
| 选项 | 说明 | 为什么不选 |
|------|------|-----------|
| **Consulting Partners** | 提供专业咨询和实施服务 | ✅ 正确答案 |
| AWS Marketplace | 在线软件商店，购买第三方软件 | ❌ 是买软件，不是买服务 |
| Technology Partners | 提供软件产品（如安全工具、监控软件） | ❌ 是产品供应商，不是服务提供商 |
| Technical Account Management (TAM) | AWS 的技术客户经理（Enterprise Support 专属） | ❌ 是 AWS 员工，不是合作伙伴 |

**记忆口诀：**
- **Consulting = 做服务（人帮你做事）**
- **Technology = 做产品（卖软件给你）**

**补充知识 - TAM：**
- Technical Account Manager 是 **Enterprise Support** 计划专属的 AWS 员工
- 提供主动指导和协助，但不是外部合作伙伴渠道

## 题目 10

**题干：** Which among the options below can you use to launch a new Amazon RDS database cluster to your VPC? (Select TWO)

**选项：**
- A. AWS CodePipeline
- B. AWS Systems Manager
- C. AWS CloudFormation ✅
- D. AWS Concierge
- E. AWS Management Console ✅

**正确答案：** C. AWS CloudFormation + E. AWS Management Console

### 知识点总结

**创建 AWS 资源的方式：**

| 方式 | 说明 |
|------|------|
| **AWS Management Console** | Web 界面，手动点击创建资源 ✅ |
| **AWS CloudFormation** | 基础设施即代码（IaC），通过模板自动化创建资源 ✅ |
| AWS CLI | 命令行工具（选项中没有） |
| AWS SDK | 编程方式调用 API（选项中没有） |
| AWS CDK | 用代码定义基础设施（选项中没有） |

**各选项分析：**
| 选项 | 说明 | 能否创建 RDS |
|------|------|-------------|
| **AWS Management Console** | AWS Web 控制台 | ✅ 可以手动创建 |
| **AWS CloudFormation** | IaC 服务，通过 YAML/JSON 模板部署资源 | ✅ 可以自动化创建 |
| AWS CodePipeline | CI/CD 管道服务，自动化代码发布流程 | ❌ 是发布流程工具，不是资源创建工具 |
| AWS Systems Manager | 运维管理服务（补丁、配置、参数存储等） | ❌ 用于管理现有资源，不是创建资源 |
| AWS Concierge | Enterprise Support 的账单和账户支持团队 | ❌ 是人工服务，不是技术工具 |

**易混淆点：**
- **CodePipeline** = CI/CD 管道（自动化部署**代码**，不是资源）
- **CloudFormation** = IaC（自动化创建**基础设施资源**）

**创建资源 vs 管理资源：**
| 目的 | 服务 |
|------|------|
| 创建资源 | Console、CloudFormation、CLI、SDK、CDK |
| 管理/运维资源 | Systems Manager |
| 自动化代码发布 | CodePipeline、CodeDeploy |

## 题目 11

**题干：** Which of the following actions will AWS charge you for?

**选项：**
- A. Network charges for the transfer of data from your data center to S3 through a VPN
- B. Provisioning elastic IPs and attaching them to running EC2 instances
- C. Transfer of EC2 files between two AWS Regions ✅
- D. Setting up additional VPCs in your account

**正确答案：** C. Transfer of EC2 files between two AWS Regions

### 知识点总结

**AWS 数据传输收费规则（重要！）：**

| 传输类型 | 收费情况 |
|----------|----------|
| **Data Transfer IN（传入 AWS）** | 🆓 **免费** |
| **Data Transfer OUT（从 AWS 传出到互联网）** | 💰 收费 |
| **跨区域（Region）传输** | 💰 **收费** |
| 同区域同 AZ 内传输（私有 IP） | 🆓 免费 |
| 同区域跨 AZ 传输 | 💰 收费 |

**各选项分析：**
| 选项 | 收费情况 | 原因 |
|------|----------|------|
| 数据中心 → S3（传入 AWS） | 🆓 免费 | Data Transfer IN 免费 |
| Elastic IP 附加到运行中的 EC2 | 🆓 免费 | 正在使用的 EIP 不收费 |
| **跨区域传输** | 💰 **收费** | 跨 Region 传输必收费 |
| 创建 VPC | 🆓 免费 | VPC 本身免费（部分组件如 NAT Gateway 收费） |

**Elastic IP 收费规则（易错！）：**

| 状态 | 收费情况 |
|------|----------|
| 附加到**运行中**的 EC2 实例 | 🆓 免费 |
| **未附加**或附加到**停止**的实例 | 💰 收费 |
| 一个实例附加**多个** EIP | 💰 额外的 EIP 收费 |

> 为什么未使用的 EIP 收费？因为 IPv4 地址是稀缺资源，AWS 鼓励你使用或释放它。

**免费服务/操作汇总：**
- 创建 VPC、子网、路由表、安全组、NACL
- Data Transfer IN（数据传入）
- 附加到运行中实例的第一个 Elastic IP
- IAM 用户、角色、策略管理

**记忆口诀：**
- **传入免费，传出收费**
- **跨区域必收费**
- **EIP 不用就要钱**

## 题目 12

**题干：** Which is a machine learning-powered security service that discovers, classifies, and protects sensitive data such as personally identifiable information (PII) or intellectual property?

**选项：**
- A. Amazon GuardDuty
- B. Amazon Cognito
- C. Amazon Rekognition
- D. Amazon Macie ✅

**正确答案：** D. Amazon Macie

### 知识点总结

**Amazon Macie：**
- 使用**机器学习**自动发现、分类和保护 **S3 中的敏感数据**
- 识别 **PII**（个人身份信息）：信用卡号、社保号、护照号、邮箱地址等
- 识别**知识产权**和其他敏感业务数据
- 生成发现结果和警报，可集成到安全工作流中

**AWS 安全服务对比（重要！）：**

| 服务 | 功能 | 关键词 |
|------|------|--------|
| **Amazon Macie** | 敏感数据发现和保护（S3） | PII、sensitive data、classify |
| **Amazon GuardDuty** | 威胁检测（恶意活动、异常行为） | threat detection、malicious activity |
| **Amazon Inspector** | 漏洞扫描（EC2、容器的 CVE） | vulnerability、CVE、security assessment |
| **AWS WAF** | Web 应用防火墙 | SQL injection、XSS、web attacks |

**题干：** Which of the following is used to enable instances in the public subnet to connect to the public Internet?

- B. NAT Gateway

### 知识点总结

**Internet Gateway (IGW)：**
- 让 **公共子网** 中的实例与互联网进行 **双向通信**

| 特性 | Internet Gateway | NAT Gateway / NAT Instance |
|------|------------------|---------------------------|
| 用于 | **公共子网** | **私有子网** |
| 流量方向 | **双向**（入站 + 出站） | **单向**（仅出站） |

```
                    Internet
                       ↕
                [Internet Gateway]
                       ↕
    ┌──────────────────┴──────────────────┐
    ↓                                      ↓
[公共子网]                            [NAT Gateway]
(有公有 IP)                               ↓
                                     [私有子网]
                                    (无公有 IP)
```

**各选项分析：**
| 选项 | 用途 |
|------|------|
| API Gateway | 创建、发布、管理 API（与互联网连接无关） |
| NAT Gateway | 让**私有子网**实例访问互联网（仅出站） |
| NAT Instance | 与 NAT Gateway 功能相同，但是自己管理的 EC2 实例 |
| **Internet Gateway** | 让**公共子网**实例与互联网双向通信 ✅ |

**记忆口诀：**
- **公共子网** + 互联网 = **Internet Gateway**
- **私有子网** + 互联网（仅出站） = **NAT Gateway**

## 题目 19

**题干：** A space agency is using Amazon S3 to store their high-resolution satellite images and videos everyday. Which of the following should they do to minimize the upload time?

**选项：**
- A. Use the Multipart upload API ✅
- B. Upload the images and videos using the BatchWriteItem API
- C. Enable Cross-Origin Resource Sharing (CORS)
- D. Shift to S3 Intelligent-Tiering storage class

**正确答案：** A. Use the Multipart upload API

### 知识点总结

**S3 Multipart Upload（多部分上传）：**
- 将大文件分成**多个部分并行上传**
- 显著**提高上传速度**（并行传输）
- 如果某个部分失败，只需**重传该部分**（不用重传整个文件）
- AWS 建议：**文件超过 100MB** 时使用多部分上传
- **必须使用**：文件超过 **5GB** 时

**Multipart Upload 优势：**
| 优势 | 说明 |
|------|------|
| 并行上传 | 多个部分同时传输，提高吞吐量 |
| 断点续传 | 失败只重传失败部分 |
| 边生成边上传 | 可以在文件生成过程中开始上传 |

**各选项分析：**
| 选项 | 分析 |
|------|------|
| **Multipart Upload API** | ✅ 并行上传大文件，减少上传时间 |
| BatchWriteItem API | ❌ 这是 **DynamoDB** 的 API，不是 S3 的 |
| CORS | ❌ 跨域资源共享配置，与上传速度无关 |
| S3 Intelligent-Tiering | ❌ 存储类优化（节省成本），与上传速度无关 |

**S3 上传方式对比：**
| 方式 | 适用场景 |
|------|----------|
| **单次上传（PUT）** | 小文件（< 100MB） |
| **Multipart Upload** | 大文件（> 100MB），必须用于 > 5GB |
| **S3 Transfer Acceleration** | 跨地理位置远距离上传加速 |

**关键词识别：**
- "large files" / "high-resolution" / "videos" + "minimize upload time" → **Multipart Upload**
- "global users" + "upload speed" → **S3 Transfer Acceleration**

## 题目 27

**题干：** You noticed that you cannot reach one of your EC2 web servers behind an ELB whenever you enter the DNS name of your load balancer. Which of the following should you first check to gain more insight on the issue?

**选项：**
- A. AWS CloudTrail
- B. AWS Config
- C. Amazon CloudWatch
- D. ELB Health Check ✅

**正确答案：** D. ELB Health Check

### 知识点总结

**为什么首先检查 ELB Health Check？**
- 如果实例**未通过健康检查**，ELB 会认为实例 **不健康**
- ELB **不会向不健康的实例转发流量**
- 这是 ELB 无法访问后端实例的**最常见原因**

**ELB Health Check 工作原理：**
```
ELB 定期检查 → 实例响应？
    ↓ 是              ↓ 否
  标记为 Healthy    标记为 Unhealthy
  转发流量给它      停止转发流量
```

**常见导致 Health Check 失败的原因：**
- 应用程序服务未运行
- 健康检查路径配置错误（如 `/health` 返回 404）
- 安全组阻止了健康检查端口
- 实例资源耗尽（CPU/内存）

**各选项分析：**
| 选项 | 用途 | 为什么不是首选 |
|------|------|---------------|
| AWS CloudTrail | 审计 API 调用 | 用于审计，不是诊断连接问题 |
| AWS Config | 记录资源配置变化 | 用于合规，不是实时诊断 |
| Amazon CloudWatch | 监控指标和日志 | 可以用，但不是第一步 |
| **ELB Health Check** | 检查后端实例健康状态 | ✅ **直接定位问题的第一步** |

**ELB 故障排查顺序：**
1. **检查 Health Check** - 实例是否健康？
2. **检查安全组** - ELB 和实例之间是否允许通信？
3. **检查目标组** - 实例是否注册到目标组？
4. **检查 CloudWatch 指标** - 更深入的监控数据

**监控和日志服务用途对比：**
| 服务 | 用途 | 关键词 |
|------|------|--------|
| **ELB Health Check** | 检查后端实例是否可用 | 连接问题、实例不可达 |
| CloudWatch | 监控指标、告警、日志 | 性能监控、阈值告警 |
| CloudTrail | 审计 API 调用 | 谁做了什么、安全审计 |
| AWS Config | 资源配置历史和合规 | 配置变更、合规检查 |

**记忆要点：**
- ELB 访问不了后端 → **首先检查 Health Check**
- Health Check 失败 = ELB 不转发流量

## 题目 28

**题干：** You are planning to deploy a video streaming application with frequently accessed, throughput-intensive workloads to your EC2 instance which requires fast, consistent throughput. What EBS volume type should you use to maximize performance as well as cost?

**选项：**
- A. General Purpose SSD
- B. Throughput Optimized HDD ✅
- C. Cold HDD
- D. Provisioned IOPS SSD

**正确答案：** B. Throughput Optimized HDD

### 知识点总结

**题目关键词分析：**
- "video streaming" → 大文件**顺序读取**
- "throughput-intensive" → 需要高**吞吐量**
- "maximize performance as well as **cost**" → 性价比优先

**EBS 卷类型对比（重要！）：**

| 卷类型 | 代码 | 特点 | 适用场景 | 成本 |
|--------|------|------|----------|------|
| General Purpose SSD | gp2/gp3 | 平衡性能和成本 | 大多数通用工作负载 | 中等 |
| **Throughput Optimized HDD** | **st1** | **高吞吐量**，大块顺序访问 | 大数据、日志、**视频流** | **低** |
| Cold HDD | sc1 | 低成本，低访问频率 | 归档、冷数据 | 最低 |
| Provisioned IOPS SSD | io1/io2 | 高 IOPS，低延迟 | 数据库、OLTP | 最高 |

**为什么选 Throughput Optimized HDD (st1)？**
1. **高吞吐量** - 最高 500 MB/s，适合视频流
2. **成本低** - HDD 比 SSD 便宜
3. **适合顺序访问** - 视频播放是大块顺序读取

**各选项分析：**
| 选项 | 分析 |
|------|------|
| General Purpose SSD | 性能够用，但成本更高 |
| **Throughput Optimized HDD** | ✅ 高吞吐量 + 低成本 = 最佳选择 |
| Cold HDD | 吞吐量低，用于冷数据 |
| Provisioned IOPS SSD | 用于高 IOPS 场景，不是吞吐量场景，成本最高 |

**IOPS vs 吞吐量（Throughput）区别：**
| 指标 | 含义 | 适用场景 |
|------|------|----------|
| **IOPS** | 每秒输入/输出操作数 | 数据库（大量小块随机读写） |
| **Throughput** | 每秒传输数据量（MB/s） | 视频流、大数据（大块顺序读写） |

**EBS 选型速记：**
| 关键词 | 选择 |
|--------|------|
| 数据库、IOPS | **Provisioned IOPS SSD (io1/io2)** |
| 通用、启动卷 | **General Purpose SSD (gp2/gp3)** |
| **吞吐量、视频、大数据** | **Throughput Optimized HDD (st1)** |
| 归档、冷数据 | **Cold HDD (sc1)** |

## 题目 32

**题干：** A customer is using Amazon S3 to store sprites of game characters. When players retrieve these sprites, they are temporarily stored on the player's computer. The sprites are currently stored in the S3 Standard storage class. Which of the following options would you recommend to optimize storage costs?

**选项：**
- A. Have the customer compress the sprites to reduce storage consumption.
- B. Add a lifecycle policy to move sprites to S3 Standard - Infrequent Access after the customer uploads them.
- C. Add a lifecycle policy to move sprites to S3 Glacier Flexible Retrieval after the customer uploads them.
- D. Have the customer directly upload the sprites to S3 Standard - Infrequent Access. ✅

**正确答案：** D. Have the customer directly upload the sprites to S3 Standard - Infrequent Access.

### 知识点总结

**题目关键分析：**
- 游戏精灵图 → 上传后**很少修改**
- 玩家检索后**缓存在本地** → 不需要重复下载
- 符合"**不频繁访问**"的特征

**为什么选 D（直接上传到 S3 Standard-IA）？**
- 精灵图的访问模式天生就是**不频繁**的
- 不需要等待生命周期策略生效
- **立即节省存储成本**

**S3 存储类对比（重要！）：**

| 存储类 | 特点 | 适用场景 | 存储成本 | 检索成本 |
|--------|------|----------|----------|----------|
| **S3 Standard** | 频繁访问 | 活跃数据 | 高 | 无 |
| **S3 Standard-IA** | 不频繁访问，即时检索 | 备份、长期存储 | 低 | 有 |
| **S3 One Zone-IA** | 单 AZ，不频繁访问 | 可重建的数据 | 更低 | 有 |
| **S3 Glacier Instant** | 归档，即时检索 | 很少访问的归档 | 很低 | 有 |
| **S3 Glacier Flexible** | 归档，检索需分钟/小时 | 归档数据 | 非常低 | 高 |
| **S3 Glacier Deep Archive** | 归档，检索需 12-48 小时 | 合规归档 | 最低 | 最高 |

**各选项分析：**
| 选项 | 分析 |
|------|------|
| 压缩精灵图 | 可以减少大小，但不是存储类优化 |
| 生命周期策略移到 IA | ❌ 需要等待，不如直接上传到 IA |
| 移到 Glacier | ❌ 检索需要时间，不适合游戏资源 |
| **直接上传到 IA** | ✅ 立即生效，数据本身就是不频繁访问 |

**S3 Standard-IA 注意事项：**
- 最小存储时间：**30 天**
- 最小对象大小：**128 KB**
- 有**检索费用**
- 适合**访问不频繁但需要即时访问**的数据

**为什么不用 Glacier？**
- Glacier Flexible Retrieval 检索需要**分钟到小时**
- 游戏资源需要**即时访问**，不能等待

**记忆要点：**
- 不频繁访问 + 需要即时检索 → **S3 Standard-IA**
- 归档 + 可以等待检索 → **S3 Glacier**

```

│ • 硬件维护和更换                   │
│ • 网络基础设施                     │

└─────────────────────────────────────┘
```

**客户负责的安全层面：**
```
┌─────────────────────────────────────┐
│         客户负责                    │
├─────────────────────────────────────┤
│ • EC2 操作系统补丁                 │
│ • 应用程序代码和配置               │
│ • 数据加密和备份                   │
│ • 安全组和 NACL 配置               │
│ • IAM 用户和权限                   │
│ • WAF、Shield 等安全服务配置       │
└─────────────────────────────────────┘
```

**常见易错点：**
- **数据复制** - 不是自动的，需要客户配置
- **WAF 保护** - 不是自动的，需要客户启用
- **OS 补丁** - EC2 上客户负责，托管服务上 AWS 负责

**记忆口诀：**
- AWS = **云的**安全（基础设施、物理、Hypervisor）
- 客户 = **云中**的安全（数据、OS、应用、配置）

**EC2 存储类型对比：**

| 存储类型 | 持久性 | 性能 | 数据保留 |
|----------|--------|------|----------|
| **Instance Store** | ❌ 临时 | **最高** | 实例停止/终止后丢失 |
| **EBS** | ✅ 持久 | 高 | 独立于实例生命周期 |

| 选项 | 分析 |

| EFS | ❌ 网络文件系统，不是临时的 |
| EBS Throughput Optimized HDD | ❌ 持久存储，不是临时的 |
| **Instance Store** | ✅ 临时 + 最高性能 |
| EBS Provisioned IOPS SSD | ❌ 持久存储，不是临时的 |

**Instance Store vs EBS：**
| 特性 | Instance Store | EBS |
|------|---------------|-----|
| 数据持久性 | ❌ 临时 | ✅ 持久 |

| 快照备份 | ❌ 不支持 | ✅ 支持 |
| 适用场景 | 缓存、临时数据 | 数据库、持久数据 |

**Instance Store 数据丢失的情况：**
- 实例停止（Stop）

- 底层硬件故障
- **注意：重启（Reboot）不会丢失数据**

**记忆口诀：**
- 看到 **"temporary"（临时）** → **Instance Store**

## 题目 44

**题干：** What are the benefits of creating snapshots of Amazon EBS volumes to back up data? (Select TWO)

**选项：**
- A. Scalability
- B. Flexibility
- C. Durability ✅
- D. Cost-Effective ✅
- E. Elasticity

**正确答案：** C. Durability + D. Cost-Effective

### 知识点总结

**EBS 快照（Snapshots）的主要好处：**

| 好处 | 说明 |
|------|------|
| **Durability（持久性）** | 快照存储在 **S3** 中，跨多个 AZ 复制，高持久性，防止数据丢失 |
| **Cost-Effective（成本效益）** | **增量备份**方式，只存储变化的块，按压缩后大小计费 |

**为什么是 Durability？**
- 快照存储在 **Amazon S3** 中
- S3 跨多个 AZ 复制数据
- 高持久性，可以在故障时恢复数据

**为什么是 Cost-Effective？**
- **增量备份** - 只存储自上次快照以来变化的块
- 不需要复制未变化的数据
- 按**压缩后的数据大小**计费
- 优化存储成本

**各选项分析：**
| 选项 | 是否是快照好处 | 说明 |
|------|---------------|------|
| Scalability（可扩展性） | ❌ | 快照与扩展资源无关 |
| Flexibility（灵活性） | ❌ | 快照不涉及资源配置的灵活性 |
| **Durability（持久性）** | ✅ | 存储在 S3，高持久性 |
| **Cost-Effective（成本效益）** | ✅ | 增量备份，节省存储成本 |
| Elasticity（弹性） | ❌ | 快照与弹性伸缩无关 |

**增量快照工作原理（Cost-Effective 的原因）：**
```
快照 1：[全量数据]
快照 2：[仅变化的块] → 指向快照 1 的未变化块
快照 3：[仅变化的块] → 指向之前快照的未变化块

→ 只存储变化的数据 = 节省存储成本！
```

**快照与 S3 的关系（Durability 的原因）：**
- 快照**底层存储在 S3**
- 享受 S3 的高持久性（99.999999999%）
- 跨多个 AZ 复制
- 但**用户无法直接访问**这些 S3 对象

**易错点 - Flexibility 为什么不对：**
- Flexibility 通常指**资源配置的灵活性**
- 虽然快照可以用于恢复、创建卷等，但这不是 AWS 定义的 "Flexibility"
- 快照的核心价值是 **Durability** 和 **Cost-Effective**

**记忆口诀：**
- EBS 快照 = **Durability**（S3 存储，高持久性）+ **Cost-Effective**（增量备份，省钱）

**题干：** Which statement is true according to Amazon RDS Multi-AZ deployment?

**选项：**
- A. Amazon RDS asynchronously creates replicates to different AZ.
- B. Amazon RDS does not support replicates for multi-AZ.
- C. Amazon RDS replicates data in a synchronous way to different AZ. ✅
- D. Amazon RDS creates replicates in both synchronous and asynchronous ways.

**正确答案：** C. Amazon RDS replicates data in a synchronous way to different AZ.

### 知识点总结

**RDS Multi-AZ 部署：**
- 使用 **同步复制（Synchronous Replication）**
- 主实例写入时，**同时**写入备用实例
- 确保数据**零丢失**
- 提供**高可用性**和**自动故障转移**

**Multi-AZ 工作原理：**
```
写入请求 → [主实例 AZ-A] ←同步复制→ [备用实例 AZ-B]
                ↓                            ↓
           确认写入                    同时确认写入
                ↓
           返回成功
```

**Multi-AZ vs Read Replica 对比（重要！）：**

| 特性 | Multi-AZ | Read Replica |
|------|----------|--------------|
| **复制方式** | **同步（Synchronous）** | **异步（Asynchronous）** |
| 目的 | 高可用性、灾难恢复 | 读取扩展、性能提升 |
| 可读取 | ❌ 备用实例不能读取 | ✅ 可以读取 |
| 故障转移 | ✅ 自动 | ❌ 需要手动提升 |
| 跨区域 | ❌ 同区域不同 AZ | ✅ 可以跨区域 |
| 数据一致性 | 强一致性 | 最终一致性 |

**各选项分析：**
| 选项 | 分析 |
|------|------|
| 异步复制 | ❌ Multi-AZ 是同步的，Read Replica 才是异步 |
| 不支持复制 | ❌ Multi-AZ 就是基于复制的 |
| **同步复制** | ✅ 正确 |
| 两种方式都用 | ❌ Multi-AZ 只用同步复制 |

**同步 vs 异步复制：**
| 复制方式 | 特点 | 数据丢失风险 |
|----------|------|-------------|
| **同步（Synchronous）** | 写入同时确认 | **零** |
| **异步（Asynchronous）** | 写入后延迟复制 | 有少量风险 |

**Multi-AZ 适用场景：**
- 生产数据库
- 需要高可用性
- 不能容忍数据丢失
- 需要自动故障转移

**记忆口诀：**
- **Multi-AZ** = **同步**复制 = 高可用 = 不能读
- **Read Replica** = **异步**复制 = 读扩展 = 可以读

## 题目 48

**题干：** A company wants to store sensitive data in an Amazon S3 bucket and encrypt it after upload. Therefore, they want to manage their own keys for encryption in AWS services. Which of the following would be used to meet this requirement?

**选项：**
- A. Customer Managed Key ✅
- B. IAM Access Key
- C. AWS Managed Key
- D. AWS Owned Key

**正确答案：** A. Customer Managed Key

### 知识点总结

**题目关键词：**
- "manage their own keys" → 自己管理密钥
- "encryption in AWS services" → 在 AWS 中加密

**AWS KMS 密钥类型对比（重要！）：**

| 密钥类型 | 管理者 | 客户控制 | 可见性 |
|----------|--------|----------|--------|
| **Customer Managed Key** | **客户** | ✅ 完全控制 | ✅ 可见 |
| **AWS Managed Key** | AWS | ❌ 有限控制 | ✅ 可见 |
| **AWS Owned Key** | AWS | ❌ 无控制 | ❌ 不可见 |

**Customer Managed Key (CMK) 特点：**
- 客户在 **KMS 中创建**
- 客户**完全控制**：
  - 定义密钥策略
  - 启用/禁用密钥
  - 设置密钥轮换
  - 删除密钥
- 可以审计密钥使用情况（CloudTrail）
- 可以跨账户共享

**各选项分析：**
| 选项 | 功能 | 为什么不选 |
|------|------|-----------|
| **Customer Managed Key** | 客户管理的 KMS 密钥 | ✅ 正确答案 |
| IAM Access Key | 编程访问 AWS 的凭证 | ❌ 不是加密密钥 |
| AWS Managed Key | AWS 管理的密钥 | ❌ 客户无法完全控制 |
| AWS Owned Key | AWS 拥有的密钥 | ❌ 客户看不到也无法控制 |

**三种 KMS 密钥详细对比：**

| 特性 | Customer Managed | AWS Managed | AWS Owned |
|------|-----------------|-------------|-----------|
| 创建者 | 客户 | AWS | AWS |
| 密钥策略 | ✅ 客户定义 | ❌ AWS 定义 | ❌ AWS 定义 |
| 轮换控制 | ✅ 客户控制 | ✅ 自动（每年） | ❌ AWS 控制 |
| 删除 | ✅ 可以 | ❌ 不可以 | ❌ 不可以 |
| 费用 | 💰 有月费 | 🆓 免费 | 🆓 免费 |
| 适用场景 | 需要完全控制 | 默认加密 | SSE-S3 等 |

**S3 加密选项：**
| 加密类型 | 使用的密钥 |
|----------|-----------|
| SSE-S3 | AWS Owned Key |
| SSE-KMS (AWS Managed) | AWS Managed Key |
| SSE-KMS (Customer Managed) | **Customer Managed Key** |
| SSE-C | 客户提供的密钥 |

**记忆口诀：**
- 想**自己管理**密钥 → **Customer Managed Key**
- 想**省事**让 AWS 管 → **AWS Managed Key**
- **IAM Access Key** ≠ 加密密钥

### 知识点总结

|-----------|

| **AWS Lambda** |

**AWS Savings Plans 类型：**

| 类型 | 覆盖范围 | 灵活性 | 折扣 |
|------|----------|--------|------|
| **Compute Savings Plans** | EC2 + Fargate + Lambda | 最高 | 最多 66% |

## 题目 50

**题干：** Which of the following are advantages of AWS managed services? (Select TWO)

**选项：**
- A. Reduced operational costs for maintaining ✅
- B. Provided free enterprise level supports
- C. Increased high-level control of infrastructure
- D. Enhanced Security aligned with your controls ✅
- E. Automatically encrypted data at rest

**正确答案：** A + D

### 知识点总结

**AWS Managed Services (AMS) 的两大优势：**

| 优势 | 详细说明 |
|------|----------|
| **Enhanced Security aligned with your controls** | AMS 构建和维护合规、运营和安全护栏，通过自动检测和修复帮助满足合规要求（HIPAA、HITRUST、GDPR、SOC、NIST、ISO、PCI、FedRAMP） |
| **Reduced operational costs** | 帮助优化 AWS 资产，客户可节省高达 **30% 运营成本**和 **25% 基础设施成本** |

**各选项分析：**
| 选项 | 分析 |
|------|------|
| **Reduced operational costs** | ✅ 节省运营和基础设施成本 |
| Free enterprise level supports | ❌ Enterprise Support **需要付费** |
| Increased control | ❌ 托管服务**减少控制**（AWS 管理底层基础设施） |
| **Enhanced Security aligned with your controls** | ✅ 自动化合规检测和修复 |
| Automatically encrypted data at rest | ❌ **AWS 不自动加密客户数据**，需要客户自己启用 |

**重要澄清 - Automatically encrypted data at rest：**
- ❌ **AWS 不会自动为客户数据加密**
- 客户需要**自己启用加密**来保护敏感数据
- 这是**客户的责任**（共享责任模型）

**Enhanced Security 的具体含义：**
- 构建合规、运营和安全**护栏（guardrails）**
- 自动检测和**修复**安全问题
- 帮助满足合规要求：HIPAA、GDPR、SOC、PCI、FedRAMP 等

**Reduced operational costs 的具体含义：**
- 财务优化整个 AWS 资产
- 节省的成本会**降低 AMS 费用**
- 按使用量付费，随时可以收回运营控制权

**易错点：**
- **自动加密是错误的** - 需要客户自己启用
- **控制权减少** - AWS 管理底层，你没有访问权限
- **支持不是免费的** - Enterprise Support 需要额外付费

**记忆口诀：**
- AMS = **安全合规**（Enhanced Security）+ **省钱**（Reduced costs）
- 加密 = **客户责任**，不是自动的

## 题目 51

**题干：** Which service should you use to get recommendations for the optimal configuration to reduce costs?

**选项：**
- A. AWS Budgets
- B. AWS Cost Explorer
- C. Amazon Redshift
- D. AWS Compute Optimizer ✅

**正确答案：** D. AWS Compute Optimizer

### 知识点总结

**AWS Compute Optimizer：**
- 分析资源**配置和使用指标**
- 推荐**优化配置**以降低成本或提高性能
- 基于**历史使用数据**（CPU、磁盘 I/O、网络 I/O）
- 例如：推荐更小的实例类型

**Compute Optimizer 支持的资源：**
| 资源类型 |
|----------|
| EC2 实例 |
| EBS 卷 |
| Lambda 函数 |
| ECS on Fargate |
| Auto Scaling Groups |

**AWS 成本管理工具对比（重要！）：**

| 服务 | 功能 | 关键词 |
|------|------|--------|
| **Compute Optimizer** | 推荐**配置优化** | recommendation, optimal configuration, right-sizing |
| **Cost Explorer** | **可视化**成本分析 | visualize, analyze, trends |
| **AWS Budgets** | 设置**预算和告警** | budget, alert, threshold |
| **Trusted Advisor** | 最佳实践**检查** | best practices, 5 pillars |
| **Cost and Usage Report** | **详细数据**导出 | granular, Athena, S3 |

**各选项分析：**
| 选项 | 功能 | 为什么不选 |
|------|------|-----------|
| AWS Budgets | 预算和告警 | ❌ 不提供配置建议 |
| Cost Explorer | 成本可视化分析 | ❌ 不提供配置建议 |
| Amazon Redshift | 数据仓库 | ❌ 与成本优化无关 |
| **Compute Optimizer** | 配置优化建议 | ✅ 正确答案 |

**Compute Optimizer vs Cost Explorer 区别：**
| 特性 | Compute Optimizer | Cost Explorer |
|------|------------------|---------------|
| 功能 | **推荐配置** | **分析成本** |
| 输出 | 实例大小建议 | 成本图表和趋势 |
| 目的 | Right-sizing | 了解花费 |

**Compute Optimizer vs Trusted Advisor 区别：**
| 特性 | Compute Optimizer | Trusted Advisor |
|------|------------------|-----------------|
| 范围 | **计算资源配置** | 多个领域（成本、安全、性能等） |
| 深度 | 深入分析使用模式 | 最佳实践检查 |
| 推荐 | 具体的实例类型 | 通用建议 |

**记忆口诀：**
- **Compute Optimizer** = 推荐**配置**（Right-sizing）
- **Cost Explorer** = **看**成本
- **Budgets** = **设**预算
